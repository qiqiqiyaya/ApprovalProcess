# ApFlow.UI 项目规则

**文件用途**：本规范用于指导 ApFlow.UI 项目的开发、维护与协作，所有开发者及协助本项目的 AI Agent 必须严格遵守。

## 一、 AI Agent 角色与任务

作为辅助本项目的 AI，你的核心任务是：

1.  **遵循规范**：生成的任何代码、文档或建议都必须符合本文件的所有规则。
2.  **理解业务**：在创建或修改组件/服务时，必须阅读或生成对应的 `*.business.md` 文件，确保理解业务上下文。
3.  **保持一致性**：在命名、代码结构、API 设计等方面与现有项目模式保持一致。
4.  **优化建议**：在遵循规范的前提下，可对代码性能、可读性、可维护性提出改进建议。

## 二、 核心技术栈

- **前端框架**: Angular 19
- **流程图库**: AntV X6
- **UI 组件库**: ng-zorro-antd 19
- **表单处理**: @delon/form
- **语言**: TypeScript 5.7 (启用严格模式)

## 三、 项目结构

### 3.1 目录布局 (`src/app/`)
app/
├── pages/ # 所有业务功能模块
│ ├── ap-flow-designer/ # 流程设计器主模块
│ ├── dynamic-form/ # 动态表单配置模块
│ ├── common/ # 跨模块共享的组件、指令、管道、工具
│ ├── models/ # 跨模块共享的数据模型与接口
│ ├── services/ # 跨模块共享的业务服务
│ └── ... # 其他业务模块
├── app.component.ts
├── app.config.ts
├── app.routes.ts
└── ...
**AI 提示**：新建功能模块应在 `pages/` 下创建独立目录。共享资源放入 `common/`, `models/`, `services/`。

### 3.2 文件命名规范（必须遵守）

- **所有文件名**：使用 **kebab-case**（短横线连接的小写字母）。
- **组件类名**：使用 **PascalCase**，并以 `Component` 结尾（Angular 惯例）。
- **示例对照**：
    - 文件：`approval-node.component.ts`
    - 类名：`ApprovalNodeComponent`
    - 选择器：`app-approval-node`
- **业务描述文件**：每个组件/模块目录内必须包含 `[组件名].business.md`。

## 四、 业务描述文件 (`*.business.md`) 规范

**AI 提示**：这是理解业务逻辑的关键，也是你必须生成或更新的文件。

每个 `*.business.md` 应包含：

1.  **业务目的**：这个组件/模块解决什么问题？
2.  **核心功能**：它具体做什么？
3.  **关键输入/输出**：`@Input()`， `@Output()`， 服务 API。
4.  **主要流程/逻辑**：简要说明代码执行的关键路径。
5.  **依赖关系**：它依赖哪些其他组件、服务或模块？
6.  **使用示例**（如适用）：简短的代码片段。

## 五、 核心编码规范

### 5.1 TypeScript

- **禁止使用 `any`**。优先使用 `interface` 定义对象结构，使用 `type` 定义联合或工具类型。
- 所有函数参数和返回值必须显式声明类型。
- 使用 `camelCase` 命名变量/函数，`PascalCase` 命名类/接口/枚举，`UPPER_SNAKE_CASE` 命名常量。

### 5.2 Angular 通用规范

- **组件**：保持小巧、单一职责。通过 `@Input()` 和 `@Output()` 与父组件通信。
- **服务**：封装业务逻辑和 HTTP 请求。使用 `providedIn: 'root'` 创建单例。方法通常返回 `Observable`。
- **生命周期钩子**：按标准顺序实现。`ngOnInit` 用于初始化，`ngOnDestroy` 用于清理订阅。
- **变更检测**：对纯展示组件优先使用 `ChangeDetectionStrategy.OnPush`。
- **路由**：使用惰性加载模块。路径使用 kebab-case。

## 六、 组件开发细则

### 6.1 文件结构

每个组件一个目录，包含：
approval-node/
├── approval-node.component.html
├── approval-node.component.css
├── approval-node.component.ts
├── approval-node.component.spec.ts
└── approval-node.business.md # 必须存在

### 6.2 通信与依赖

- **父子通信**：严格使用 `@Input()` 和 `@Output()`。
- **跨组件/无关组件通信**：通过共享的 **Service**（利用 `BehaviorSubject` / `Signal`）实现，**禁止**使用 `EventEmitter` 在非父子组件间穿透。
- **依赖注入**：通过构造函数注入服务，而不是在组件内部实例化。

### 6.3 样式 (CSS)

- 使用 Angular 默认的模拟封装 (`ViewEncapsulation.Emulated`)。
- 采用 **BEM** 命名范式（`.block__element--modifier`）以减少冲突。
- **禁止**在组件样式中使用 `!important`。

### 6.4 AntV X6 图形开发（特殊要求）

- **注册与管理**：自定义节点/边应在 `common/x6-shapes/` 下的专用服务中集中注册。
- **操作封装**：所有对 `Graph` 实例的操作（添加、删除、连线）必须封装在服务中（如 `GraphOperationService`），组件调用服务方法。
- **初始化时机**：X6 Graph 的初始化必须在组件的 `ngAfterViewInit` 生命周期钩子中进行。

## 七、 服务开发细则

- **命名**：服务类名使用 `PascalCase`，并以 `Service` 结尾（例如 `FlowDataService`）。
- **HTTP 调用**：使用 `HttpClient`。统一错误处理和超时设置应在拦截器中实现。
- **状态管理**：使用 `BehaviorSubject` 或 Angular `Signal` 管理应用状态。仅在状态非常复杂时考虑引入 NgRx。

## 八、 测试与文档

### 8.1 单元测试 (`*.spec.ts`)

- **目标**：核心业务逻辑覆盖率 > 80%，组件 > 70%。
- **位置**：测试文件与被测文件同级。
- **要求**：使用 `TestBed` 配置。模拟所有外部依赖（服务、路由）。

### 8.2 代码注释

- **公开 API**：为所有 `public` 和 `protected` 的方法、属性编写 JSDoc 注释。
- **复杂逻辑**：在代码块上方用简短注释说明 **“为何这样做”**，而非描述代码本身。
- **AI 提示**：你生成的代码应自带清晰注释。

## 九、 开发与部署流程

- **代码检查**：提交前必须通过 `ng lint`。
- **提交信息**：遵循 Conventional Commits，例如：`feat(flow-designer): add parallel approval node`。
- **构建命令**：
    - 开发：`npm start`
    - 生产构建：`npm run build:prod`
- **环境配置**：`environment.development.ts` 与 `environment.production.ts`。

## 十、 给 AI Agent 的特别注意事项

1.  **生成文件名时**：务必使用 **kebab-case**。这是最常见的问题点。
2.  **类型安全**：在任何情况下都避免使用 `any`。如果类型暂时复杂难以定义，可先用 `unknown` 并标注 `TODO`。
3.  **业务先行**：在动手写代码前，先询问或确认该组件的 `*.business.md` 内容，确保理解业务场景。
4.  **性能意识**：在生成涉及循环（`*ngFor`）的模板时，主动建议添加 `trackBy` 函数。对于纯展示组件，建议使用 `ChangeDetectionStrategy.OnPush`。
5.  **可维护性**：优先推荐组合模式而非继承。一个函数/方法只做一件事。
6.  **问题诊断**：如果遇到关于 X6 图形渲染、变更检测或异步数据流的问题，请优先查阅本规范第 11 节（常见问题）。
